## 3. 整體資料流

```
Overpass API
    │  (1) fetch_osm_data: ~14k OSM nodes
    ▼

    │
    ▼
build_embed_text  →  Gemini gemini-embedding-001, output_dim=1536 (batch≤100, sleep 0.5s)

    ▼

    │
    ▼
```

設計重點：**所有寫入 DB 都走「先比對已存在 osm_id」+ `ON CONFLICT DO NOTHING`** 雙保險，讓 script 100% idempotent，可以隨時 re-run。

## 5. 關鍵設計決策

### 5.1 ORM 中怎麼處理 `vector(1536)` 與 `TEXT[]`
- `vector` 用 `pgvector.sqlalchemy.Vector(1536)`。
- `tags` 用 `sqlalchemy.dialects.postgresql.ARRAY(Text)`。
- `osm_id` 標 `unique=True`，方便程式端 query「已存在的 osm_id」時走 unique index。
- 繼承 M0 提供的 `Base`（`from app.database import Base`），不自建 metadata。

### 5.2 `parse_osm_element` 的細節判斷
- **名稱優先順序**：`tags['name']`（OSM 慣例：local language，台北就是中文）→ `tags['name:zh']` → 沒有就 return None。
- **`name_en`**：`tags['name:en']`，沒有就 None。
- **category**：依 spec 的 `TAG_TO_CATEGORY`，按 `amenity → tourism → leisure → shop` 順序找第一個命中的。
  - 注意 `market` 在 spec 對應 `food`（不是 attraction），照表抄即可。
- **description**：把 `cuisine`、`opening_hours`、`wheelchair`、`phone`、`website` 等存在的 tag 用「; 」串起來；都沒有就 None。
- **tags 欄位**：`[primary_tag_value, cuisine_value]`，後者沒有就只放一個。
- **address**：OSM 原生 address 很零碎，先簡單 join `addr:full` 或 `addr:street + addr:housenumber`，沒有就 None。M3 那邊可以再 geocoding 補。
- **try/except 包整段**：任何 KeyError / TypeError / ValueError 直接 `return None` 跳過該筆，不讓壞資料炸掉整批。

### 5.3 `build_embed_text` 的字串組合
照 spec：`"{name}。{category}。{address or ''}。{description or ''}。{' '.join(tags)}"`
中文句號「。」當分隔符讓 embedding 模型能識別欄位邊界。

### 5.4 `batch_embed` 與 `embedder.embed_batch` 的關係（一個小歧義）
spec M1.2 在 script 裡寫 `async def batch_embed(texts, client: AsyncOpenAI)`，但 spec M3.3 又把 `embed_batch(texts)` 列為 `services/embedder.py` 的 frozen signature（M1 / M3 / M5 共用）。

**處理方式**：以 `services/embedder.py:embed_batch` 為單一真相，script 內的 `batch_embed` 就是**薄 wrapper / 直接 forward**。
- M3 / M5 共用 batching 邏輯（rate limit、retry）一處維護
- script 行為仍與 spec 描述一致（每 100 筆、間隔 0.5s）
- 既然全 repo 已切到 Gemini，spec 那個 `client: AsyncOpenAI` 參數型別自動失效；wrapper 不需要 client 參數

### 5.5 Gemini embedding 細節（取代 spec 原本的 OpenAI）
- **SDK**：`google-genai==0.8.0`，`from google import genai`、`client = genai.Client(api_key=...)`、`await client.aio.models.embed_content(...)`。
- **模型**：`gemini-embedding-001`。
  - 原生輸出 3072 dim，**必須**帶 `output_dimensionality=1536` 才能塞進 `vector(1536)` schema。
  - MRL 截斷後的 1536 維 embedding 需要 L2 normalization 再寫入，否則 cosine similarity 會偏（Google 文件明示）。
- **task_type**：
  - 灌庫（M1）→ `RETRIEVAL_DOCUMENT`
  - 之後查詢（M5）→ `RETRIEVAL_QUERY`
  - 兩種要配對使用。
- **Batch 上限**：Gemini `embed_content` 一次最多 100 筆，剛好對齊 spec。
- **失敗處理**：429 / 5xx → sleep 5s 重試一次；再失敗整批拋 RuntimeError（在 ingest 主流程才能決定要 abort 還是降級）。
- **Client lifecycle**：module-level lazy init（第一次 call 才建），避免重複建 HTTP session。

### 5.6 寫入策略
- 用 SQLAlchemy 的 `pg_insert(Attraction).values([...])` 一次塞 200 筆，加 `.on_conflict_do_nothing(index_elements=['osm_id'])`。
- 每個 chunk 一個 transaction，失敗只影響該 chunk。
- session 用 M0 提供的 `from app.database import SessionLocal`，`async with SessionLocal() as session`。

### 5.7 錯誤處理
- Overpass 偶爾 429 / 504：手寫簡單重試（最多 2 次，間隔 30s），不引入 tenacity 以免動 requirements。再失敗就 `RuntimeError` 結束。
- Gemini rate limit / 5xx：見 5.5。
- `/tmp/*.mp4`、`/tmp/*.jpg` 清理：M1 沒產出這些檔案，spec 寫的「清除暫存」可能是複製貼上，這邊保留 no-op 的 `_cleanup_tmp()`，註解說明這是給 M2 的鉤子。

# 第二回合：M1 ↔ M5 嫁接設計

> §1–§8 是 M1 第一輪實作的歷史紀錄（其中 §5.5 對 Gemini embedding 的選型已被 §9 取代）。
> 這一段是 M1 第二輪 + M5 的整合設計。

## 9. 為什麼從單 embedder 改成雙 embedder

第一輪用 bge-m3 padded 1024→1536 跑通了 ingest，但 M5 要做的是「user 的 places 的 centroid → 在 attractions 找最相似」的檢索任務，單一 model 召回容易偏：bge-m3 在中文長語意強但對地名/品牌名相對弱，mpnet 反之。決定改成**雙路檢索 + RRF 合併**。

| Model | params | native dim | 強項 |
|-------|--------|-----------|------|
| `BAAI/bge-m3` | ~560M | **1024** | 中文長句、語意檢索 |
| `sentence-transformers/paraphrase-multilingual-mpnet-base-v2` | ~110M | **768** | 中文短語、地名/品牌 |

兩個都 L2-normalize，**不再做 padding**，各自存 native dim 的欄位，各自一條 ivfflat（cosine）index。

## 10. 為什麼要做 bulk embedding dump

15k+ rows × 2 個 model 在 CPU 上跑完約 4–6 小時。隊友本機沒 GPU，docker 啟動就要等半天才能 query 是不可接受的。

**解法**：本機 GPU 跑一次完整 ingest → `pg_dump --data-only -t attractions -Fc` → 上傳 Hugging Face dataset → 隊友 docker 啟動時 bootstrap script 偵測 `attractions` 為空就拉 dump 還原。Query-time embedding（單一 query 句子）仍跑在容器內 model，不痛。

## 11. Schema 變更（migration `0002_dual_embeddings.py`）

兩張表都要動。**`places` 屬於 M3，但 schema 改變不影響 frontend / API contract，M3 owner 那邊只需改 `routers/places.py` 的 4 行 embed call**。

```sql
-- attractions
DROP INDEX IF EXISTS attractions_embedding_ivfflat;
ALTER TABLE attractions DROP COLUMN IF EXISTS embedding;
ALTER TABLE attractions ADD COLUMN embedding_bgem3 vector(1024);
ALTER TABLE attractions ADD COLUMN embedding_mpnet vector(768);

-- places
DROP INDEX IF EXISTS places_embedding_ivfflat;
ALTER TABLE places DROP COLUMN IF EXISTS embedding;
ALTER TABLE places ADD COLUMN embedding_bgem3 vector(1024);
ALTER TABLE places ADD COLUMN embedding_mpnet vector(768);
```

ivfflat index **故意不在 migration 內建立**：pgvector 文件明示 ivfflat 要在「已有資料」時建效果才好（list centroids 從現有 vector 取樣）。索引交給 `ingest_osm.py` 在 phase 2 結束時 `CREATE INDEX IF NOT EXISTS ...`，dump 會帶上索引定義；隊友 `pg_restore` 自動拿到。`places` 是 per-session 小表，先不建 index（順序掃即可），未來如果某 session > 1000 筆再說。

## 13. Query 流程（澄清 reels 是何時 embed 的）

> **Reels 不是在 `/recommend` 才 embed**。Reels embedding 發生在「user 把 reels 透過 Line 加入收藏」那一刻（M3 `POST /places`）。`/recommend` 完全不再 embed 任何東西。

完整資料流：

```
[ingest 階段，本機 GPU]
    OSM/觀光署/Wikidata
        │
        ▼ ingest_osm.py phase 2: gather(embed_batch_bgem3, embed_batch_mpnet)
    attractions.embedding_bgem3 / embedding_mpnet  ← 本機算好
        │
        ▼ pg_dump → HF dataset → 隊友 pg_restore
    隊友 docker 內 attractions 同步狀態


[使用者收藏階段，per place 一次]
    Line user 丟 Reels 連結
        │
        ▼ M2 process_reels_url（解析 metadata）
        ▼ M3 routers/places.py:create_place
        ▼ embed_text = f"{name}。{category}。{description or ''}"
        ▼ gather(embed_bgem3, embed_mpnet)
    INSERT places (..., embedding_bgem3, embedding_mpnet)


[推薦查詢階段，user 點「推薦」]
    POST /recommend { session_id, category, limit }
        │
        ▼ SELECT * FROM places WHERE user_session_id = :s
        │     已含兩欄 embedding，零 embedding cost
        │
        ▼ centroid_bge   = np.mean(places.embedding_bgem3, axis=0)   # 1024
        │ centroid_mpnet = np.mean(places.embedding_mpnet, axis=0)   # 768
        │
        ▼ 並行兩條 pgvector ANN：
        │   SELECT id,...,embedding_bgem3 <=> :c_bge AS dist
        │     FROM attractions WHERE category=:cat
        │     ORDER BY dist ASC LIMIT 20
        │   (同上 mpnet)
        │
        ▼ RRF 合併：score(a) = Σ 1/(60 + rank_i(a))
        │   k=60 是文獻默契值；只要 a 同時被兩個 model 排在前面就會被推上去
        │   取合併後 top-{limit}
        │
        ▼ asyncio.gather() 並行呼叫 Gemini 2.5 flash
        │   每個結果一個短句，個別失敗 fallback 成預設文案，不讓整 endpoint 500
        │
        ▼ return RecommendResult[]
```

關鍵點：**`/recommend` 端點 CPU-only 也能 < 1s**（瓶頸是 Gemini 5 個並行請求），因為沒有任何 embedding 推論。

## 14. RRF 合併邏輯細節

```python
def rrf_merge(rankings: list[list[int]], k: int = 60) -> list[tuple[int, float]]:
    """
    rankings: 多個 model 各自 top-N 的 attraction id list（rank 從 0 開始）。
    回傳 [(attraction_id, fused_score)] 依 score DESC。
    """
    scores: dict[int, float] = {}
    for ranking in rankings:
        for rank, aid in enumerate(ranking):
            scores[aid] = scores.get(aid, 0.0) + 1.0 / (k + rank + 1)
    return sorted(scores.items(), key=lambda x: x[1], reverse=True)
```

每個 model 取 top-20 進 RRF，融合後切 top-`limit` 給 Gemini。

`score` 欄位回給前端用「兩 model 平均的 cosine distance」（直覺好理解，數值小越好），不是 RRF score：

```python
avg_dist = (dist_bge_per_id.get(id, 1.0) + dist_mpnet_per_id.get(id, 1.0)) / 2
```

### 15.2 Bootstrap（隊友容器啟動時）

`scripts/bootstrap_attractions.py`：
1. `SELECT count(*) FROM attractions` → 若 > 0，直接 return
2. `huggingface_hub.hf_hub_download` 拉最新 dump 到 `/tmp/attractions.dump`
3. `subprocess.run(["pg_restore", "--data-only", "-d", DATABASE_URL, dump])`
4. 印 sanity check（總數、`embedding_bgem3 IS NOT NULL` 比例）

`docker-compose.yml` backend service 加 entrypoint：

```yaml
entrypoint:
  - bash
  - -c
  - |
    alembic upgrade head &&
    python scripts/bootstrap_attractions.py &&
    uvicorn app.main:app --host 0.0.0.0 --port 8000
```

### 15.3 版本對齊保護

dump 檔名帶 `g{shortSHA}` + embedder 兩個 model name 的 sha256[:8]。bootstrap 對當前 `embedder.py` 算同樣 hash → 對不上就 abort 並印警告。避免 model 換了還用舊 vector 跑出垃圾結果。
